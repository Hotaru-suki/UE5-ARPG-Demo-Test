## 3. 详细缺陷记录

### BUG-001 多 AI 场景下怪物会互相伤害

**问题现象**

在高密度 AI 战斗场景中，敌人之间会互相造成伤害。随着 AI 数量增加，该现象更明显，并进一步触发死亡、掉落和对象累积。

**复现条件**

- 场景内同时存在多个可接敌 AI
- AI 已进入战斗状态
- 存在范围伤害或多目标技能行为

**复现步骤**

1. 进入压力测试场景。
2. 批量生成多个 AI。
3. 让多个 AI 在相近区域同时进入战斗。
4. 观察受击反馈、血量变化和死亡情况。

**预期结果**

- 怪物只对合法敌对目标造成伤害。
- 同阵营对象不应互相进入伤害结算。

**实际结果**

- 怪物之间会互相造成伤害。
- 连带影响死亡、掉落和性能统计结果。

**影响范围**

- 战斗规则正确性
- AI 压力测试结果可信度
- 掉落与对象数量统计
- 性能趋势解释

**原因分析**

项目中虽然存在阵营或队伍概念，但没有将其统一接入目标收集和伤害过滤链。目标一旦被收集，就可能直接进入效果应用和伤害结算阶段。

**修复建议**

- 在目标收集阶段统一过滤同阵营目标。
- 在伤害应用前增加二次合法性校验。
- 将阵营规则从“有定义”推进到“参与结算判断”。

### BUG-002 技能目标过滤规则不统一

**问题现象**

不同技能或不同目标类型之间，目标合法性规则缺少统一过滤层，存在行为不一致风险。

**复现条件**

- 项目中存在多个技能入口
- 不同技能采用不同 TargetType 或蓝图目标链路

**复现步骤**

1. 触发不同技能或不同目标类型的攻击行为。
2. 对比命中对象和命中范围。
3. 观察是否存在不该命中的对象被带入结算。

**预期结果**

- 同类技能应遵循统一目标过滤规则。
- 无效对象不应进入结算阶段。

**实际结果**

- 目标过滤行为存在分散实现风险。
- 技能扩展后更容易出现规则漂移。

**影响范围**

- 技能命中一致性
- 蓝图维护成本
- 新技能扩展稳定性
- 测试复现稳定性

**原因分析**

项目缺少统一的目标有效性过滤层，部分规则分散在 Ability、TargetType 或蓝图资源中实现，导致不同链路之间容易出现实现偏差。

**修复建议**

- 建立统一目标过滤入口。
- 明确敌我、死亡状态、可攻击性等判断顺序。
- 降低核心规则对单个蓝图实现的依赖。

### BUG-003 装备槽位写入缺少严格合法性校验

**问题现象**

当前装备写入接口存在状态被写坏的风险，理论上可能出现槽位类型和物品类型不匹配、无效物品进入槽位等情况。

**复现条件**

- 背包系统与装备槽位系统可正常使用
- 角色已拥有可装备物品

**复现步骤**

1. 获取多个不同类型物品。
2. 对不同槽位执行装备、替换、卸下操作。
3. 检查背包、槽位和角色状态是否一致。

**预期结果**

- 只有合法物品才能进入合法槽位。
- 背包数据和装备状态保持一致。

**实际结果**

- 当前实现存在校验不足风险。
- 异常状态理论上可被写入并持续存在。

**影响范围**

- 背包与装备一致性
- 角色状态正确性
- UI 展示准确性
- 后续测试结果可信度

**原因分析**

写入接口对物品持有关系、物品类型和目标槽位合法性校验不足，导致状态边界没有被严格约束。

**修复建议**

- 写入前统一校验物品持有关系、物品类型和槽位合法性。
- 非法写入直接拒绝并输出日志。
- 为装备和卸下流程补充回归测试。

### BUG-004 存档恢复存在未持有物品仍处于装备状态的风险

**问题现象**

读取存档时，存在“背包里不持有，但槽位里仍处于已装备状态”的风险。

**复现条件**

- 项目支持存档和读档
- 已进行拾取、装备、卸下等操作

**复现步骤**

1. 在当前状态下执行存档。
2. 退出并重新加载流程。
3. 检查背包内容、装备槽位和角色状态。
4. 对多种物品状态重复验证。

**预期结果**

- 读档后背包和装备状态完全一致。
- 不应存在未持有却装备的物品。

**实际结果**

- 当前逻辑存在此类异常状态风险。
- 数据异常或边界条件下更容易暴露。

**影响范围**

- 存档可信度
- 角色状态一致性
- 背包与装备联动结果
- 功能回归测试稳定性

**原因分析**

存档恢复更偏向直接恢复槽位内容，而不是基于背包持有关系重新做一致性校验，因此一旦读入脏数据，异常状态就可能被保留下来。

**修复建议**

- 读档后增加背包与装备一致性校验。
- 非法装备状态自动剔除或重置。
- 对异常恢复情况输出明确日志。

### BUG-005 地图边缘或石块附近击杀怪物后，怪物会异常飞起

**问题现象**

在地图边缘、石块附近或局部碰撞较复杂的位置击杀怪物时，怪物死亡后会出现异常飞起或明显不自然的位移表现。

**复现条件**

- 敌人靠近地图边缘、障碍物或石块
- 在该位置被击杀
- 死亡时触发受击、击退、物理或动画过渡

**复现步骤**

1. 将怪物引导到地图边缘或石块附近。
2. 在贴近障碍或边缘的位置将怪物击杀。
3. 观察尸体或死亡表现的位移情况。
4. 重复多次验证是否稳定复现。

**预期结果**

- 怪物死亡后应正常倒地或进入稳定死亡表现。
- 不应出现明显飞起、弹飞或异常位移。

**实际结果**

- 特定场景位置下，怪物会异常飞起。
- 表现与周围碰撞环境存在明显关联。

**影响范围**

- 战斗表现质量
- 角色与怪物死亡反馈自然度
- 边界场景稳定性
- 录屏和展示观感

**原因分析**

高概率与以下因素叠加有关：

- 边缘区域碰撞处理
- 石块或障碍物附近的碰撞体交叠
- 死亡、受击、击退或物理切换时的瞬时速度处理
- 动画与物理表现切换时未完全收敛

**修复建议**

- 排查死亡瞬间的碰撞体与速度状态。
- 检查边缘和障碍物附近的碰撞设置。
- 对死亡表现与物理切换阶段增加约束。

### BUG-006 AutoPlay 入口缺少稳定代码接口，自动化接入需要保留人工触发

**问题现象**

项目存在 AutoPlay 能力入口，但当前更稳定的触发方式仍然依赖蓝图输入链路，自动化测试中需要保留人工按一次 `BackSpace` 的步骤。

**复现条件**

- 使用当前自动化测试脚本启动流程
- 需要接入项目现有 AutoPlay 行为

**复现步骤**

1. 启动自动化脚本。
2. 进入场景。
3. 尝试完全依赖代码触发 AutoPlay。
4. 对比人工按 `BackSpace` 与代码触发的稳定性。

**预期结果**

- AutoPlay 应具备稳定、可直接调用的代码接口。
- 自动化流程不应依赖人工触发。

**实际结果**

- 当前更稳定的方案仍然是手动按一次 `BackSpace`。
- 纯代码侧难以无损接管现有蓝图输入链路。

**影响范围**

- 自动化完整度
- 回归流程可重复性
- 自动化接入成本
- 蓝图与代码协作边界清晰度

**原因分析**

AutoPlay 入口主要掌握在蓝图和输入资源侧，缺少稳定公开的 C++ 调用接口。代码侧即使模拟输入，也难以完全替代现有蓝图链路的触发方式。

**修复建议**

- 为 AutoPlay 增加稳定 C++ 公开接口。
- 将关键启动逻辑从输入资源侧下沉到可调用层。
- 在不破坏现有蓝图资源的前提下逐步收口入口。
